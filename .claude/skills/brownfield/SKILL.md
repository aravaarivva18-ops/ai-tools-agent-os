# brownfield

Шаблон по умолчанию greenfield: PRD с нуля → всё растёт из него. В brownfield
**код уже есть и он — правда о системе**. В openspec нет нативного `onboard`, поэтому
делаем той же машинерией, но с адаптированным входом.

## Что сдвигается против greenfield

1. **`project.md` заполняется ИЗ кода**, а не из головы — это главный бутстрап-шаг.
2. **`north-star.md` → инварианты становятся щитом:** в первую очередь *что нельзя
   сломать* (публичный API, схема БД, auth, форматы данных), см. [[north-star]].
3. **`PRD.md` = scoped-бриф** конкретной работы (фича/рефактор/эпик), а не полный
   продуктовый документ. Опиши в нём только то, что делаешь сейчас.

**Бутстрап (один раз на репо):** `openspec init`, затем агент читает кодовую базу и
заполняет `project.md` + `north-star.md`. Промпт даёт `./scripts/start.sh --brownfield`.

## Три грабли brownfield (проверь ДО цикла)

1. **Пред-существующие красные гейты → цикл заклинит на первой итерации.** Прогони
   `./scripts/gates.sh` на нетронутом репо. Красный (чужой старый lint/тест падает) —
   почини отдельным prep-коммитом ДО авто-блока, иначе агент бьётся в чужую ошибку.
   (`start.sh --brownfield` делает эту проверку и предупреждает.)
2. **`MODIFIED`/`REMOVED` дельты требуют baseline-спеки.** Меняешь существующее
   поведение — сначала дай агенту записать текущее поведение затрагиваемой области в
   `openspec/specs/<capability>.md`, чтобы дельте было что модифицировать. Для чистого
   `ADDED` не нужно.
3. **Полный прогон тестов каждую итерацию на большом репо медленный.** Сузь `gates.sh`
   до затронутой области (или прими цену многочасовых прогонов).

## Три типа работы

### Фича — добавить новое в существующее
Ложится в поток напрямую. `explore` тут даже полезнее — читает код, находит точки
интеграции. Дельта = `ADDED Requirements`. Baseline-спеки не нужны. Инварианты
north-star стерегут старое, пока добавляешь новое.

### Рефактор — внутренности меняются, поведение НЕТ
Самый хитрый: нового требования нет → дельта почти пустая («MODIFIED: внутренняя
структура, поведение сохранено»). Критерий «готово» = **старое поведение не изменилось**:
- сначала убедись, что область покрыта тестами; нет — добавь характеризующие тесты
  отдельным маленьким change ПЕРЕД рефактором;
- тогда `gates.sh` (существующие тесты) = страховка рефактора, ровно для этого нужен
  детерминированный гейт (см. [[verify-loop]]);
- инвариант в north-star: «поведение и публичный контракт `<area>` не меняются»;
- `tasks.md` = механические шаги (extract/rename/move), каждый держит тесты зелёными.

### Эпик — большой, много фич
НЕ делай одним гигантским change. В OpenSpec нет объекта «эпик» — эпик = **набор
связанных changes**:
- бриф эпика в `PRD.md` (или секции north-star) → несколько папок в
  `openspec/changes/`, каждая размером с фичу, в порядке зависимостей ([[plan-tasks]]);
- гоняй `./scripts/loop.sh --all` (или `goal-run.sh --all`) — идёт по changes по очереди;
- north-star несёт сквозную цель эпика + инварианты через все changes → они не
  разъезжаются (защита от дрейфа окупается здесь);
- archive каждой по мере готовности.

## Композиция

Хочешь утверждать каждый шаг — совмести с [[interactive-flow]]:
`./scripts/start.sh --brownfield --interactive`. Гейты те же, но GATE 0/1 берут
контекст из кода. Общая карта — [[spec-loop]].
