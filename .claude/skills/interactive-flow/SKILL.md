# interactive-flow

Противоположность «схлопнутому» бутстрапу (`start.sh` по умолчанию делает всё до
proposal одним заходом). Здесь агент идёт **по одному этапу за раз** и на каждой
границе жёстко останавливается на гейте.

## Железное правило

**Ни один этап не начинается, пока предыдущий не получил явное «OK».** Агент не
«проскакивает» дальше, даже если уверен. На каждом гейте человек может:
- `OK` — этап принят, идём дальше;
- `правки: <что изменить>` — агент правит артефакты этого этапа и **снова показывает гейт** (не идёт вперёд);
- `стоп` — выход из флоу.

Формат гейта — показывай ровно так (чек-лист = что человек должен прочитать до решения):

```
═══ GATE <N> · <ЭТАП> — нужно твоё решение ═══
Сделано: <1-2 строки, что появилось/изменилось на этом этапе>
Прочитай перед решением:
  [ ] <путь к файлу> — <что там>
  [ ] <путь к файлу> — <что там>
Дальше НЕ иду без явного «OK».
Ответь:  OK  ·  правки: <...>  ·  стоп
```

## Этапы и гейты (greenfield)

- **GATE 0 · PRD** — человек заполнил (или агент помог заполнить) `openspec/PRD.md`.
  Чек-лист: `openspec/PRD.md`. → жди OK.
- **GATE 1 · КОНТЕКСТ** — агент вывел из PRD `openspec/project.md` (стек/конвенции) и
  `openspec/north-star.md` (цель + инварианты, см. [[north-star]]).
  Чек-лист: `project.md`, `north-star.md`. → жди OK.
- **GATE 2 · EXPLORE** — агент прогрокал интент первой фичи (`/opsx:explore` или
  нативно), показал план и варианты в диалоге. Файла может не быть — показывай план текстом.
  Чек-лист: план в диалоге (+ затронутые файлы, если читал код). → жди OK.
- **GATE 3 · PROPOSE** — агент создал change (`proposal.md`, `design.md`, `tasks.md`,
  `specs/delta.md`). Это критический архитектурный гейт. Здесь чек-лист = **ВСЕ
  документы проекта**, чтобы человек мог перечитать целиком перед автономией:
  ```
    [ ] openspec/PRD.md
    [ ] openspec/north-star.md
    [ ] openspec/project.md
    [ ] openspec/changes/<name>/proposal.md
    [ ] openspec/changes/<name>/design.md
    [ ] openspec/changes/<name>/tasks.md
    [ ] openspec/changes/<name>/specs/delta.md
  ```
  → жди OK.
- **→ AUTO** — только после OK на GATE 3 запускается авто-блок
  (`./scripts/goal-run.sh <name>` / `loop.sh <name>`). Дальше работает [[verify-loop]].

## Жёсткие гейты (машинное принуждение)

Гейты — не только дисциплина в промпте. В интерактивном режиме (`start.sh --interactive`
ставит маркер) работает PreToolUse-хук `require-gate-approval.sh`: **пока человек не
одобрил `propose`, запись КОДА физически заблокирована** — Write/Edit по исходникам
вернёт ошибку, писать можно только спеки/доки/планы. Разблокировка — только человеком:

```
./scripts/gate.sh ok propose      # спросит подтверждение с терминала; агент так не может
./scripts/gate.sh status          # что одобрено, включён ли режим
./scripts/gate.sh reset           # перед новой фичей
```

На GATE 3 после «OK» скажи человеку выполнить `./scripts/gate.sh ok propose` — без этого
авто-блок не сможет писать код, даже если ты попробуешь. Одобрять сам себя не пытайся
(файл одобрений хук тоже защищает от записи агентом).

## Важно

- Не создавай код и не отмечай `tasks.md` до OK на GATE 3.
- `tasks.md` разбивай по [[plan-tasks]] — на GATE 3 человек проверяет и гранулярность.
- Этот протокол композится с [[brownfield]]: там те же гейты, но GATE 0/1 берут
  контекст из существующего кода, а не из PRD с нуля.
- Общая карта всего цикла — [[spec-loop]].
