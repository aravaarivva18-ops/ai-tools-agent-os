# north-star

`openspec/PRD.md` — исходный продуктовый документ (точка входа №1, читается на
бутстрапе). `project.md` — про стек и конвенции. `openspec/specs/` — про то, что
система умеет ПРЯМО СЕЙЧАС (дескриптивно). `openspec/north-star.md` — про то, КУДА
идём (прескриптивно) и что ни один change не имеет права сломать по дороге.

North-star — это **дистиллят PRD**: цель и инварианты, сжатые из большого PRD в
короткую версию, которую не жалко читать каждой итерацией. PRD → (бутстрап) → north-star.

## Зачем это отдельный слой

В прогоне `./scripts/loop.sh --all` (или `goal-run.sh --all`) каждый change видит
только свой `proposal.md`/`design.md`. Общий продуктовый интент не переносится ничем
→ дрейф копится от change к change: каждый локально «зелёный», а вместе они уводят
продукт в сторону. North-star — единственный файл, который несёт цель и границы
через ВСЕ change'ы. Он читается каждой итерацией (см. [[spec-loop]], `PROMPT.md`).

## Как заполнять

Держи коротким — это «конституция намерения», не роадмап:

1. **Цель** — одно предложение. Если change ей не служит, он под вопросом.
2. **Определение успеха** — 2-4 наблюдаемых критерия (не задачи).
3. **Инварианты** — границы, которые нельзя ломать. Формат: правило + (если проверяемо)
   как оно проверяется. Примеры: «деньги только через `billing/`», «публичный API
   обратно совместим», «секреты только из env».
4. **Вне north-star** — что сознательно НЕ строим (чтобы цикл туда не уходил).

## Инвариант → гейт (главное)

Прозаический инвариант держится на дисциплине агента и легко нарушается тихо.
**Всё, что можно проверить кодом — выноси в `scripts/architecture-check.sh`** (это Gate 4
в `gates.sh`). Тогда нарушение инварианта краснит гейт и блокирует коммит, а не только
«просится» в промпте. Это разница между «агент обещал не трогать billing» и
«коммит физически не пройдёт, если он тронул».

Примеры fitness-функций уже заготовлены в `architecture-check.sh` — раскомментируй и
подгони под свои INV-*. Проверяй только изменённые файлы (`git diff`), это дёшево.

## Когда обновлять

- При старте проекта — заполнить один раз.
- Когда появляется новая граница, которую нельзя нарушать → новый INV + (по возможности) его проверка.
- Если цикл третий раз упирается в конфликт со спекой — часто это значит, что change
  противоречит инварианту; не переписывай инвариант молча, зови человека (см. [[verify-loop]]).
