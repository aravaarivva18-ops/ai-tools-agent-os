# plan-tasks

`tasks.md` — не список «что сделать», а очередь для авто-блока: цикл берёт ПЕРВЫЙ
неотмеченный `[ ]`, делает ровно его, гейтит, коммитит, отмечает `[x]`, идёт дальше
(см. [[spec-loop]], [[verify-loop]]). Значит качество декомпозиции = стабильность цикла.

## Критерий хорошего пункта

Каждый пункт должен быть **маленьким и проверяемым изолированно** — как в TDD:
можно реализовать и прогнать `gates.sh` за один проход агента.

- ✅ «добавить поле `email` в модель User + миграция»
- ✅ «валидировать формат email в `createUser`, тест на невалидный»
- ❌ «сделать авторизацию» — слишком крупно, агент утонет, гейты не сойдутся
- ❌ «отрефакторить весь модуль» — нет чёткого «готово», ревью не сможет судить

Признак «слишком крупно»: не можешь назвать конкретную проверку, которая станет
зелёной, когда пункт закрыт. Если проверки нет — пункт не готов к циклу.

## Формат

Нумеруй группами (`1.1`, `1.2`, `2.1`) — логические блоки, внутри мелкие шаги.
Порядок = порядок выполнения (цикл идёт сверху вниз). Зависимые шаги — ниже тех,
от кого зависят.

```
- [ ] 1.1 <мелкий шаг, проверяемый изолированно>
- [ ] 1.2 <следующий>
- [ ] 2.1 <новая логическая группа>
```

## Если пункт всё равно велик

Агенту разрешено (и предписано в `PROMPT.md`) **разбить крупный пункт на подпункты
прямо в `tasks.md`** перед тем как брать его, и взять первый подпункт. Но лучше
разбить руками на этапе плана — тогда каждый коммит остаётся атомарным, а откат
(`git tag` на задачу в `loop.sh`) — гранулярным.

## Чего не делать

- Не пиши пункты, требующие архитектурного решения — это должно быть решено в
  `design.md` ДО tasks (иначе агент угадает, а угадывать он не должен).
- Не пакуй несколько несвязанных изменений в один пункт — ревью [[verify-loop]]
  завернёт как scope creep.
- Не оставляй «доделать позже» / TODO-пункты как способ обойти полноту реализации.
