# spec-loop

Этот проект не кодирует без openspec change. Полный цикл жизни фичи — два блока.

## Блок 0 — бутстрап проекта (один раз, до всего)

Точка входа №1 — **`openspec/PRD.md`**. Это первый инпут харнесса: человек кладёт
туда исходный продуктовый документ. Если PRD ещё не разобран:

1. Прочитай `openspec/PRD.md`.
2. Выведи из него `openspec/north-star.md` (цель + инварианты, см. [[north-star]]) и
   `openspec/project.md` (стек, конвенции).

PRD читается здесь и в `/opsx:explore`, но НЕ в каждой итерации авто-блока (он большой —
там работает дистиллят `north-star.md`). Всё дальше растёт из PRD.

## Ручной блок = одно одобрение плана на старте (дефолт: автономный режим)

Дефолт харнесса — «пообщались на старте → оставили на дни/недели → эскалация только в
крайних случаях» (AGENTS.md прав. 7). Значит manual-фаза проходит ОДИН раз на весь PRD,
а не по одному change с остановками.

1. **Explore → Propose ВЕСЬ PRD.** Декомпозируй PRD в набор changes: на каждую крупную
   фичу `/opsx:explore` (прогрокать интент, поймать «строим не то») → `/opsx:propose`
   (или `./scripts/new-change.sh <name>` + natural language, если нет слэш-команд).
2. **Plan.** Каждый `tasks.md` разбей на маленькие проверяемые пункты (см. [[plan-tasks]]).
3. **Одно одобрение.** Покажи человеку ПЛАН ЦЕЛИКОМ (все changes) один раз. Его запуск
   `goal-run.sh --all` / `loop.sh --all` = одобрение. Дальше НЕ останавливайся на
   одобрение каждого change — иди подряд, эскалируй только на крайних блокерах. Строгий
   per-этапный режим с гейтами (одобрять каждый шаг) — опционально, [[interactive-flow]].

## Auto-блок (агент решает, КАК реализовать, до зелёных гейтов)

3. **Loop.** Как только `tasks.md` готов и одобрен — выбери один из двух путей:
   - `./scripts/goal-run.sh <name>` — нативный `/goal` в Claude Code, проще, одна
     сессия на change. Для всего проекта разом: `./scripts/goal-run.sh --all`.
   - `./scripts/loop.sh <name>` — bash-драйвер, работает и с Codex/OpenCode,
     git tag на каждую отдельную задачу. Для всего проекта: `./scripts/loop.sh --all`.
   Не путай `/goal` (толкает к условию) и `/loop` (перезапуск по таймеру для
   мониторинга) — для "доделай пока не зелёно" нужен именно `/goal`.

4. **Review + Gates.** Перед коммитом каждой задачи — независимое ревью (не самопроверка):
   subagent `verifier` внутри claude-сессии, а `loop.sh` вдобавок гоняет кросс-модельное
   ревью через `codex exec` (ревьюит НЕ та модель, что писала код). Затем — зелёный
   `./scripts/gates.sh`: финальный детерминированный hard halt (включает Gate 4 —
   инварианты `north-star.md`). LLM-ревью идёт ПЕРЕД гейтами, не вместо них.

## Снова ручной блок (человек принимает результат)

5. **Archive.** Когда все пункты `tasks.md` отмечены `[x]` — НЕ архивируй
   автоматически. Но это и **не повод останавливать автономный прогон**: переходи к
   следующему change, а завершённые оставь готовыми к архиву. Archive — батч-шаг
   человека на возврате (`/opsx:archive <name>` по каждому), а не гейт посреди недельного
   прогона. Финальную сводку «что готово к archive» положи в `fix_plan.md`.

## Когда НЕ использовать этот скилл

Для тривиальных однострочных правок (опечатка, комментарий, форматирование)
полный цикл propose→apply→archive избыточен — просто сделай правку напрямую
и предупреди пользователя, что пропустил openspec ради мелочи.
