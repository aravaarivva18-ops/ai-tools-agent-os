# verify-loop

«Готово» не решает та же модель, что писала код. Два независимых слоя, строго в
этом порядке — ревью ПЕРЕД гейтами, не вместо них.

## Слой 1 — независимое LLM-ревью (не самопроверка)

Модель, которая писала код, плохой ревьюер своей работы. Поэтому:

- **В claude-сессии** — вызывай subagent `verifier` (`.claude/agents/verifier.md`) на
  `git diff HEAD` против активного пункта `tasks.md`, `design.md` и инвариантов
  [[north-star]]. Он отвечает строго `PASS`/`FAIL` с причинами, кода не пишет.
- **В `loop.sh`** — вдобавок гоняется кросс-модельное ревью на уровне bash: `codex exec`
  ревьюит diff (или `claude`, если исполнитель — codex). Ревьюит НЕ та модель, что
  писала → независимее самопроверки. Коммит только на `REVIEW: PASS`.

Ревью ловит то, что гейты не видят: заглушки/TODO вместо логики, scope creep,
несоответствие пункту `tasks.md`, удалённые ради прохода тесты, нарушенные инварианты.

**Ревьюер ВСЕГДА прогоняет `./scripts/fallow-review.sh`** — детерминированный анализатор
TS/JS [fallow](https://github.com/fallow-rs/fallow) (dead code, дубли, сложность, границы
архитектуры, гигиена зависимостей, PR-risk verdict) — и вплетает его находки в отчёт:
`verdict=fail` → сильный сигнал к FAIL, `warn` → отражается в причинах. Это даёт ревью
детерминированную опору вдобавок к суждению модели. Оба слоя ревью (subagent `verifier` и
codex/claude в `loop.sh`) получают вывод fallow.

Если вердикт FAIL — почини по замечаниям и перепрогони, **не коммить**.

## Слой 2 — детерминированные гейты (финальный hard halt)

`./scripts/gates.sh` — единственный источник истины «готово ли». Решает скрипт, а не
агент. Порядок: дешёвое первым (lint/types), дорогое последним (tests/security/architecture).

- `--quick` — только lint/types, для PostToolUse-хука (быстрый сигнал между правками).
- Полный прогон — на Stop-хуке и в `loop.sh`/`goal-run.sh` перед коммитом.
- **Fail-closed:** если не отработала ни одна НАСТОЯЩАЯ проверка (lint/typecheck/tests) —
  это FAIL, а не молчаливый PASS. Ненастроенный проект не зелёный. Security и
  architecture-гейты сами по себе не делают код верифицированным.
- Gate 4 = `architecture-check.sh` — машинные инварианты [[north-star]].

Подгони команды под свой стек руками (`pnpm`, кастомные скрипты) — это обычный bash.
Прогони `./scripts/gates.sh` один раз ДО автоматики: если падает на пустом месте
(тесты вообще не настроены) — почини сейчас, иначе цикл будет упираться в это вечно.

## Порядок в цикле

```
реализовал пункт → ревью (слой 1, PASS) → gates.sh (слой 2, зелёный) → commit → следующий пункт
```

Гейты красные третий раз подряд на одном месте → остановись, запиши в `fix_plan.md`
→ «Известные проблемы», зови человека. Обычно это неполный/противоречивый `design.md`
или конфликт с инвариантом — чини спеку, не код. См. общий поток в [[spec-loop]].
