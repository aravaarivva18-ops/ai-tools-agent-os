## 🎯 1. Введение и архитектура скиллов
**BuilderIO/skills** — это библиотека компактных, компонуемых текстовых навыков (инструкций) для кодинг-агентов. Проект создан Builder.io для повышения дисциплины ИИ в критических точках: планирование, ревью, контроль лимитов и поиск актуальной документации.

### Архитектурная структура скилла:
Каждый навык размещается в отдельной директории в `skills/` и содержит:
1.  **`SKILL.md`:** Основной файл инструкций с YAML-фронтматером (свойства `name` и `description` для триггер-матчинга) и Markdown-инструкциями для модели.
2.  **`README.md`:** Краткое описание для разработчика.
3.  **`references/` (опционально):** Дополнительные детальные спецификации, примеры и шаблоны.

> [!NOTE]  
> Архитектура BuilderIO на 100% совпадает с архитектурой плагинов и кастомизаций Antigravity (YAML + Markdown в `SKILL.md`), что делает интеграцию технически бесшовной.

## ⚖️ 2. Анализ соответствия стандартам v3.2 (Fit Analysis)

### 2.1. Composability & Latency (Идеально)
*   **Плюс:** Скиллы являются чисто текстовыми (Markdown) инструкциями. Они имеют **нулевой оверхед по производительности** (0ms latency при запуске) и минимальный расход токенов.
*   Это полностью соответствует нашему требованию по скорости старта системы **<2s**.

### 2.2. Solo Loop Compliance (Частично)
*   **Соответствуют Solo Loop:**
    *   `/read-the-damn-docs` — заставляет агента осуществлять веб-поиск по официальной документации вместо гадания по памяти. Полезно для TDD и Source-Driven Development.
    *   `/quick-recap` — стандарт цветовых сигналов завершения ответа (🟢Done, 🟡Pending, 🔴Blocked) для прозрачности логов.
    *   `/plow-ahead` — регламент принятия консервативных решений при мелких неясностях без остановки выполнения.
    *   `/stay-within-limits` — превентивная проверка лимитов токенов во избежание сбоев посреди длинных сессий.
*   **Нарушают Solo Loop (Отклонить):**
    *   `/efficient-fable` и `/plan-arbiter` — требуют деления задач между дорогой моделью-планировщиком (Fable) и дешевыми моделями-исполнителями. В Strict Solo Loop v3.1 это запрещено (overengineering).

### 2.3. Визуальное планирование (`/visual-plan`, `/visual-recap`)
*   Эти скиллы генерируют богатые интерактивные планы в формате MDX, требующие внешнего рендерера (hosted app от Agent-Native).
*   **Наш вердикт:** Внедрение MDX-рендерера и стороннего хостинга избыточно (bloatware). Вместо этого мы адаптируем их стандарты (Mermaid-диаграммы, разметка файлов, списки открытых вопросов) внутри наших локальных Markdown-артефактов в `brain/<id>/`.

## 🛠️ 3. Рекомендации по адаптации для Antigravity

Мы принимаем гибридную стратегию: **адаптируем лучшие концепты без перетягивания внешних CLI-зависимостей.**

1.  **Интеграция `/read-the-damn-docs`:**
    *   Адаптировать регламент Docs-First в наши правила [AGENTS.md](file:///Users/rus/.gemini/config/AGENTS.md) и [CLAUDE.md](file:///Users/rus/ai-tools/CLAUDE.md). При работе с внешними API (например, Stripe, Next.js, 1C-Bitrix) агент обязан сначала выполнять веб-поиск через `agent-reach`.
2.  **Внедрение `/quick-recap`:**
    *   Принять формат короткого цветового статуса в самом конце каждого turn-ответа в CLI. Это упростит восприятие логов пользователем (предпринимателем, не разработчиком).
3.  **Исключение мультиагентных скиллов:**
    *   Игнорировать `/efficient-fable`, `/efficient-frontier` и `/plan-arbiter` по правилу YAGNI.
