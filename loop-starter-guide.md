# Как пользоваться loop-starter — гайд для тех, кто видит это впервые

Этот файл — самостоятельная инструкция. Не нужно знать ничего про loop
engineering, OpenSpec или agent harness'ы заранее — все термины объясняются
по ходу дела, там, где встречаются первый раз.

---

## Что это вообще такое

**Harness** (харнесс) — это обвязка вокруг AI-агента (Claude Code, Codex,
OpenCode), которая решает за него: что делать дальше, как проверять
результат, когда остановиться. Сам агент — просто "мозг", который пишет код
по запросу. Харнесс — это правила и скрипты вокруг него, которые превращают
разовые запросы в управляемый процесс.

`loop-starter` — конкретный харнесс из трёх слоёв:

1. **OpenSpec** — слой "что строим". Прежде чем агент напишет хоть строчку
   кода, появляется документ: зачем это нужно, что именно меняется, из каких
   маленьких шагов состоит. Без этого документа агент не имеет права
   кодить.
2. **Gates** (гейты) — слой "как проверяем, что готово". Автоматические
   проверки (линтер, типы, тесты) решают, можно ли считать шаг выполненным.
   Не агент решает "я закончил", а скрипт, который либо проходит, либо нет.
3. **Auto-блок** — слой "кто жмёт повторить, пока не done". Скрипт, который
   гоняет агента по одной маленькой задаче за раз, автоматически, без
   твоего участия — пока весь список задач не станет зелёным.

Аналогия: ты — тимлид, который написал джуну чёткое ТЗ (OpenSpec), джун
работает сам, но обязан показать зелёные тесты перед каждым коммитом
(gates), и вместо того чтобы стоять у него над душой, ты запускаешь скрипт,
который сам тыкает джуна "следующая задача" (auto-блок), пока список не
кончится.

## Что должно быть готово перед стартом

Проверь по порядку в терминале:

```bash
git --version      # любая версия
node --version     # нужен для установки OpenSpec
npm --version
claude --version    # Claude Code CLI — должен быть установлен и авторизован
```

Если `claude --version` не находит команду — сначала установи и залогинь
Claude Code, харнесс без него не заработает (или используй `codex`/`opencode`
как альтернативный агент — см. дальше).

Также понадобится сам архив `loop-starter.zip` — он должен быть рядом с этим
файлом или прислан отдельно.

## Шаг 1 — распаковать харнесс в проект

```bash
cd /путь/к/твоему/проекту
unzip -o loop-starter.zip -d /tmp/ls
cp -rn /tmp/ls/loop-starter/. .
chmod +x scripts/*.sh .claude/hooks/*.sh
```

`-n` в `cp` значит "не перезаписывать существующие файлы" — если в проекте
уже был свой `CLAUDE.md` или `.claude/settings.json`, они не пострадают,
скопируются только новые файлы. `chmod +x` возвращает право на запуск
скриптам — иногда слетает при распаковке zip.

**Если у тебя уже был git-репозиторий без единого коммита** — сделай хотя бы
один (`git add -A && git commit -m "init"`), скрипты харнесса опираются на
`git status`/`git commit`, без начального коммита некоторые команды
споткнутся.

---

## Шаг 2 — свести конфиги (если уже был свой AI-конфиг)

Если у тебя уже есть свой `CLAUDE.md` или `.claude/settings.json` — открой
файлы из архива рядом и перенеси руками:

- из `AGENTS.md` — секцию "Как мы здесь работаем" (правила: одна задача за
  раз, без заглушек-имитаций кода, самообновление конфига) добавь в свой
  `CLAUDE.md` как отдельный раздел
- из `.claude/settings.json` — блок `hooks` добавь в свой `settings.json`,
  не замени файл целиком

Если своего конфига раньше не было — просто ничего не делай, файлы из
архива уже на месте.

## Шаг 3 — установить OpenSpec и инициализировать

```bash
npm install -g @fission-ai/openspec
openspec init
```

⚠ Ставь именно `@fission-ai/openspec`. Голый пакет `openspec` в npm — чужая
заглушка (версия 0.0.0, без бинаря); `npm install -g openspec` поставит мусор,
и `openspec init` выдаст `command not found`.

Спросит, под какие инструменты настраивать (выбери Claude Code, и Codex/
OpenCode, если ими тоже пользуешься). После этого в проекте появится
рабочая структура `openspec/` — папка, где будут жить все "спеки" (описания
того, что должна делать система).

Что такое **спека**: не код, а текстовое описание требования — что система
обязана делать, при каких условиях, как это проверить. `openspec/specs/` —
это набор таких описаний, которые считаются "текущей правдой" о системе
прямо сейчас.

## Шаг 4 — положить PRD и раскрутить из него контекст

**Точка входа №1 — `openspec/PRD.md`.** Это официальный первый инпут: сюда
кладётся твой продуктовый документ (что строим, для кого, цели, ключевые
фичи, ограничения). Шаблон уже лежит по этому пути — просто заполни разделы.
Именно тут харнесс ищет старт: путь `openspec/PRD.md` прописан в reading-list
(AGENTS.md/CLAUDE.md) и в скилах, агент знает про него в начале каждой сессии.

Дальше в интерактивной сессии агента (открой `claude` в папке проекта и напиши):

```
Прочитай openspec/PRD.md и заполни из него:
- openspec/project.md — стек, конвенции, что нельзя трогать без разрешения
- openspec/north-star.md — цель продукта и инварианты (границы, которые ни одна
  фича не имеет права сломать)
```

Это разовая настройка на весь проект (не на отдельную фичу):

- `project.md` — агент читает в начале каждой сессии, чтобы не выяснять заново
  стек и правила.
- `north-star.md` — **дистиллят PRD**: цель + инварианты. Читается КАЖДОЙ
  итерацией (PRD для этого слишком большой). Это защита от дрейфа: в прогоне
  `--all` каждая фича видит только свой change, а north-star несёт общий интент
  через все change'ы. Машинно-проверяемую часть инвариантов вынеси в
  `scripts/architecture-check.sh` — тогда их ловит гейт, а не только промпт.

Почему PRD отдельно от north-star: PRD — большой исходник, читается только на
бутстрапе и в `explore`; north-star — сжатая версия, которую не жалко тащить в
контекст каждой итерации. PRD меняется редко; сдвинулось направление продукта —
правь PRD, потом синхронизируй north-star.

## Шаг 5 — создать первую фичу ("change")

**Change** — это папка с описанием одной конкретной фичи: зачем она нужна,
что меняется, из каких шагов состоит. Пока change не создан и не одобрен —
агент не имеет права писать код по этой фиче.

Сначала — **`/opsx:explore`** (нативный этап OpenSpec, не пропускай):

```
/opsx:explore <что хочешь построить своими словами>
```

Здесь агент читает код, взвешивает варианты и формирует план — это точка,
где ловится «строим не то», ДО того как всё зафиксировано в спеке. Только
после explore превращай план в формальный change:

```
/opsx:propose <имя-фичи>
```

(если слэш-команды не подхватились — просто напиши текстом: "создай
openspec-предложение для <фичи>")

Появится папка `openspec/changes/<имя-фичи>/` с четырьмя файлами:

- **`proposal.md`** — зачем это нужно и что меняется (без деталей реализации)
- **`design.md`** — как технически это сделать (архитектура, подход)
- **`tasks.md`** — чек-лист маленьких шагов реализации
- **`specs/delta.md`** — что именно должно измениться в поведении системы,
  формально (в формате "дано / когда / тогда")

## Шаг 6 — прочитать и одобрить ПЛАН ЦЕЛИКОМ (один раз, это важно)

Дефолт харнесса — автономный: ты общаешься с агентом и одобряешь план **один раз
на старте**, а дальше он работает сам дни/недели и дёргает тебя только в крайних
случаях. Поэтому агент декомпозирует **весь PRD** в набор changes сразу, а не по
одному с остановками.

Открой `proposal.md` и `design.md` каждого предложенного change и прочитай.
**Это твоя единственная точка контроля** — дальше агент их не пересматривает и не
спрашивает по каждому отдельно. Не нравится подход в каком-то `design.md` — поправь
текстом в файле сейчас или скажи агенту в диалоге, до запуска автоматики.

Также глянь `tasks.md`: каждый пункт маленький и проверяемый ("добавить поле X в
модель" — хорошо; "сделать авторизацию" — плохо, слишком крупно).

> Хочешь одобрять КАЖДЫЙ шаг, а не только план на старте — это отдельный режим:
> `./scripts/start.sh --interactive` (гейт после каждого этапа, см. ниже).

## Шаг 7 — подогнать проверки под свой стек

```bash
cat scripts/gates.sh
```

**Gate** (гейт) — это автоматическая проверка, которая либо пропускает
изменение, либо нет. `gates.sh` — заготовка, которая сама пытается угадать
твой стек (смотрит на `package.json`/`pyproject.toml` и ищет команды
`lint`/`test`/`typecheck`). Если у тебя нестандартные команды (`pnpm`
вместо `npm`, кастомный скрипт тестов) — поправь `gates.sh` руками, это
обычный bash-файл.

Прогони один раз вручную, ДО того как включать автоматику:

```bash
./scripts/gates.sh
```

Если падает на пустом месте (например, тесты ещё не настроены в принципе) —
почини это сейчас. Автоматика в шаге 9 будет упираться в ту же ошибку
бесконечно, если её не устранить заранее.

---

## Шаг 8 — понять разницу между двумя командами: `/goal` и `/loop`

Прежде чем запускать автоматику, важно не перепутать эти две команды Claude
Code — они звучат похоже, но делают разное:

| | `/goal` | `/loop` |
|---|---|---|
| Что делает | доводит работу до проверяемого условия, ход за ходом | перезапускает один и тот же промпт **по таймеру** |
| Когда останавливается | когда отдельная модель-проверяющий подтвердит "готово" | никогда сама — это для мониторинга ("проверяй раз в 5 минут") |
| Годится для "доделай пока не всё зелёное" | ✅ да | ❌ нет — не сходится к результату |

Харнесс использует именно `/goal` (через готовый скрипт, см. ниже) — не
`/loop`. Если когда-нибудь захочешь написать команду вручную, а не через
скрипт — бери `/goal`.

## Шаг 9 — первый прогон автоматики, вручную, на одной фиче

**Не начинай сразу с "прогнать весь проект".** Первый раз запусти на
одной фиче с небольшим лимитом, чтобы посмотреть вживую, как это работает:

```bash
./scripts/goal-run.sh <имя-фичи> --turns 15
```

Что произойдёт: скрипт скажет агенту "работай, пока весь `tasks.md` не
станет отмечен `[x]`, и последний прогон `gates.sh` не завершится успешно, и
всё не будет закоммичено — или остановись через 15 ходов". Агент будет
брать пункты `tasks.md` по одному, реализовывать, гонять `gates.sh`,
коммитить, переходить к следующему.

Смотри, что происходит на экране. Проверяй: берёт ли по одному пункту за
раз, реально ли гоняет проверки, не пытается ли менять
`openspec/specs/` напрямую (это запрещено правилами харнесса — там должна
появляться только финальная, проверенная человеком версия).

## Шаг 10 — наблюдение и что делать, если что-то пошло не так

```bash
tail -f logs/loop.log      # что происходит прямо сейчас
git log --oneline -10       # какие шаги уже закоммичены
cat fix_plan.md              # если агент сам застрял — он пишет причину сюда
```

Если гейты падают на одном и том же месте три раза подряд — по встроенным
правилам харнесса агент обязан **сам остановиться** и написать причину в
`fix_plan.md`, а не пытаться бесконечно. Если видишь, что цикл завис или
делает что-то не то — просто останови процесс (`Ctrl+C` в терминале, где
запущен `goal-run.sh`), незакоммиченные изменения останутся в рабочей
директории, ничего не потеряется.

---

## Шаг 11 — когда всё зелёное: архивация

Когда все пункты `tasks.md` отмечены `[x]`, автоматика сама **не**
переносит изменения в `openspec/specs/` — это финальный шаг, который
специально оставлен ручным, чтобы ты проверил результат перед тем, как он
станет "официальной правдой" о системе:

```
/opsx:archive <имя-фичи>
```

После этого `specs/delta.md` этой фичи сливается в `openspec/specs/`, а
папка change переезжает в `openspec/changes/archive/`.

---

## Шаг 12 — когда освоился: весь проект сразу

Если фич (changes) уже несколько и ты доверяешь автоматике после первых
ручных прогонов — можно гонять все сразу одной командой:

```bash
./scripts/goal-run.sh --all --turns 40
```

Это пройдёт по очереди по всем папкам в `openspec/changes/`, у которых есть
`tasks.md`, и доведёт каждую до зелёного состояния. Archive всё равно
остаётся ручным шагом на каждую фичу отдельно.

Если вместо Claude Code используешь Codex или OpenCode, либо хочешь
git-тег на каждую отдельную задачу (не только на всю фичу целиком) — есть
второй вариант автоматики, `loop.sh`, с тем же принципом:

```bash
./scripts/loop.sh <имя-фичи> --agent claude
./scripts/loop.sh --all --agent opencode --keep-going
```

## Два дополнительных режима старта

Всё выше — greenfield одним махом. Есть ещё два режима (флаги `start.sh`, композятся).

### Brownfield — код уже есть (`--brownfield`)

Когда проект не пустой и ты добавляешь **фичу**, делаешь **рефактор** или тянешь
**эпик** поверх существующего кода, который нельзя ломать:

```bash
./scripts/start.sh --brownfield
```

Отличия от greenfield:

- **Контекст берётся из кода**, а не из PRD с нуля — агент читает кодовую базу и
  заполняет `project.md` + `north-star.md`. `PRD.md` тут = короткий бриф текущей работы.
- **Инварианты north-star = что нельзя сломать** (публичный API, схема БД, auth).
- **Тип работы важен:** фича → обычный поток; рефактор → критерий «поведение не
  изменилось» + опора на существующие тесты (нет покрытия — сначала добавь тесты);
  эпик → не один огромный change, а несколько связанных + `--all`.
- **Грабля №1** (скрипт проверяет сам): если гейты на нетронутом репо уже красные —
  почини это ДО авто-блока, иначе цикл будет биться в чужую ошибку.

Детали — скил `.claude/skills/brownfield/`.

### Пошаговый с утверждением (`--interactive`)

Когда хочешь контролировать каждый шаг, а не отдавать весь ручной блок одним промптом:

```bash
./scripts/start.sh --interactive
```

Агент идёт по одному этапу за раз (PRD → контекст → explore → propose) и на каждой
границе показывает **GATE** с чек-листом документов к прочтению, а дальше **не идёт без
твоего явного `OK`**. На гейте можно ответить `правки: ...` — агент поправит и покажет
гейт снова.

Причём гейты **машинно-принудительные**: пока ты не одобрил `propose`, запись кода
физически заблокирована хуком (агент может писать только спеки/доки). Разблокируешь ты,
из терминала:

```bash
./scripts/gate.sh ok propose     # спросит [y/N] — агент так одобрить не может
./scripts/gate.sh status         # что одобрено, включён ли режим
./scripts/gate.sh reset          # перед новой фичей
```

Детали — скил `.claude/skills/interactive-flow/`. Режимы совмещаются:
`./scripts/start.sh --brownfield --interactive`.

## Частые проблемы

**`gates.sh` падает сразу, ничего не успев проверить**
Скорее всего, не нашёл команду `test`/`lint` в `package.json` — открой
`scripts/gates.sh` и пропиши свои команды руками (это обычный bash-скрипт,
не магия).

**`git commit` внутри скрипта ничего не делает**
Значит агент не внёс изменений за эту итерацию — проверь `logs/loop.log`,
возможно застрял на непонятной задаче. Разбей соответствующий пункт
`tasks.md` на более мелкие подпункты руками.

**`claude: command not found`**
Claude Code CLI не установлен или не в `PATH`. Без него сработает только
`loop.sh` с флагом `--agent codex` или `--agent opencode`, если они у тебя
настроены — `goal-run.sh` требует именно `claude`.

**Автоматика третий раз подряд падает на одном месте**
Это специально встроенное поведение, не баг: харнесс запрещает агенту
бесконечно биться в одну и ту же стену. Открой `fix_plan.md` — там должна
быть записана причина. Обычно это значит, что сама спека (`design.md`)
неполная или противоречивая — почини её руками, не код.

**Хочу остановить прогон посреди работы**
`Ctrl+C` в терминале, где запущен `goal-run.sh`/`loop.sh`. Незакоммиченные
изменения останутся на диске — ничего не потеряется, гейты просто не
дадут их закоммитить в сломанном виде.

## Мини-глоссарий

- **Харнесс** — обвязка вокруг AI-агента: правила, скрипты, проверки.
- **Change** — папка с описанием одной фичи (`proposal.md`, `design.md`,
  `tasks.md`, `specs/delta.md`), пока не заархивирована.
- **Спека / spec** — формальное описание того, что система должна делать.
  `openspec/specs/` — набор текущих, уже подтверждённых спек.
- **Gate (гейт)** — автоматическая проверка (тест/линтер/типы), которая
  решает, можно ли считать шаг выполненным.
- **Auto-блок** — часть харнесса, которая гоняет агента по задачам без
  участия человека, пока не кончится список или лимит.
- **PRD** — `openspec/PRD.md`: исходный продуктовый документ, точка входа №1.
  Первый инпут харнесса; из него выводятся north-star и changes. Читается на
  бутстрапе и в explore, не в каждой итерации.
- **North Star** — `openspec/north-star.md`: дистиллят PRD — цель продукта +
  инварианты, которые нельзя ломать ни одним change. Читается каждой итерацией —
  защита от дрейфа.
- **`/opsx:explore`** — нативный этап OpenSpec ПЕРЕД propose: прогрокать интент,
  взвесить варианты, поймать «строим не то» до формализации.
- **`/goal`** — команда Claude Code: работай, пока не выполнится условие.
- **`/loop`** — команда Claude Code: перезапускай промпт по таймеру
  (для мониторинга, не для "доделай спеку").
- **Archive** — ручной финальный шаг: перенос проверенной фичи из
  `changes/` в `specs/` как новой текущей правды о системе.
- **Brownfield** — режим для существующего кода (`start.sh --brownfield`): контекст
  из кода, инварианты = что нельзя сломать; фича/рефактор/эпик.
- **GATE** — в интерактивном режиме точка остановки после этапа: агент показывает
  чек-лист документов и ждёт `OK`. Машинно принудителен — код заблокирован до
  одобрения `propose`.
- **`gate.sh`** — команда человека для одобрения гейтов (`ok propose` / `status` /
  `reset`). Агент одобрить сам не может (нужен терминал).
