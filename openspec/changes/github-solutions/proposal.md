# Предложение: Интеграция Repository Map (по мотивам Aider/Cline)

## Исследованные решения с GitHub
1. **smolagents (CodeAgent)** — выполнение ИИ-кода в изолированной Python-среде.
   * *Вердикт*: Не использовать на текущем этапе (оверхед для нашей YAGNI-системы).
2. **Mem0 (Memory graph)** — сложная база знаний фактов.
   * *Вердикт*: Не использовать ( Obsidian RAG + task_observer полностью перекрывают эти задачи локально).
3. **Repository Mapping (Aider)** — генерация компактной карты кодовой базы (структура каталогов, классы, методы, сигнатуры функций) для предоставления ИИ-агенту мгновенного понимания архитектуры.
   * *Вердикт*: **Сделать легкую внутреннюю реализацию**.

## Предлагаемое решение
Создать легковесный модуль `tools/repo_mapper.py`, который:
1. Сканирует питоновские файлы в `tools/` и `core/`.
2. Извлекает структуру классов, сигнатуры функций и краткие описания (docstrings) с использованием встроенной библиотеки Python AST (Abstract Syntax Trees) без внешних зависимостей.
3. Генерирует компактный текстовый файл `vault/repo_map.txt` (или `.md`), описывающий структуру проекта.
4. Подключает этот файл в планировщик `PlanningWithFiles` при прогреве сессии.

## Ожидаемый эффект
- Снижение количества ходов на исследование codebase перед планированием на 50-70%.
- Мгновенное понимание архитектуры агентом при старте сессии за минимальное количество токенов.
