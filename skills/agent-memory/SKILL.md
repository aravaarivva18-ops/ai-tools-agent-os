# Адаптивная и Долгосрочная Память Агентов (Memory Engine)

Этот навык описывает стандарты построения систем памяти ИИ-агентов, позволяющих им накапливать знания о пользователе, сохранять контекст сессий и самообучаться на основе опыта.

---

## 🏛️ 1. Трехуровневая структура памяти (Memory Scoping)

Память агента делится на три изолированные области видимости:

1. **User Scope (Память пользователя):** Глобальные факты, привычки и предпочтения пользователя, сохраняющиеся между всеми сессиями (например, «Пользователь предпочитает темную тему», «Пользователь пишет на Python»).
2. **Session Scope (Память сессии):** Контекст текущего диалога или конкретной задачи. Сбрасывается при завершении сессии.
3. **Agent Scope (Память агента):** Внутренние поведенческие настройки самого агента, шаблоны успешного выполнения задач и результаты промпт-тестов.

---

## 🧠 2. Алгоритм консолидации памяти (Decision Engine)

Вместо сохранения полной истории переписки (что приводит к раздуванию контекста и лишним затратам API-токенов), сообщения сжимаются до **атомарных фактов** (Atomic Facts).

При поступлении новой информации движок памяти (Decision Engine) принимает одно из 4 решений относительно базы знаний:
*   **`ADD`:** Добавить новый факт в базу (если информации о факте ранее не было).
*   **`UPDATE`:** Обновить существующий факт (если поступили уточняющие или изменившиеся данные).
*   **`DELETE`:** Удалить старый факт (если новая информация полностью опровергает или отменяет старый факт).
*   **`NONE`:** Проигнорировать (если факт уже есть в базе в том же виде).

## 💾 3. SQL-схема хранения фактов (Causal Graph Schema)

Для связи фактов и отслеживания истории изменений используется следующая схема таблиц (подходит для SQLite и PostgreSQL):

```sql
-- Таблица для хранения текущих активных фактов памяти
CREATE TABLE agent_memory_facts (
    id VARCHAR(255) PRIMARY KEY,
    user_id VARCHAR(255),
    session_id VARCHAR(255),
    scope VARCHAR(50) NOT NULL, -- 'user', 'session', 'agent'
    fact_text TEXT NOT NULL,      -- Пример: "User prefers Go over Python"
    metadata JSONB,              -- Дополнительные теги
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Таблица аудита и истории изменений (для самообучения)
CREATE TABLE agent_memory_history (
    id SERIAL PRIMARY KEY,
    fact_id VARCHAR(255) REFERENCES agent_memory_facts(id) ON DELETE CASCADE,
    operation VARCHAR(20) NOT NULL, -- 'ADD', 'UPDATE', 'DELETE'
    old_value TEXT,
    new_value TEXT,
    rationale TEXT,                 -- Обоснование от LLM, почему факт изменен
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 🔍 4. Оптимизация поиска (Hybrid Search)

При поиске фактов памяти перед отправкой запроса в LLM выполняется гибридный поиск:
1. **Семантический поиск (Vector Search):** Векторная БД (например, pgvector) сканирует базу фактов по косинусному сходству с текущим вопросом пользователя.
2. **Фильтрация по метаданным (Metadata Filtering):** Результаты фильтруются по `user_id` и `scope` на уровне базы данных перед тем, как попасть в контекст системного промпта модели.
3. **Ограничение контекста:** В промпт передаются только топ-N наиболее релевантных фактов, что минимизирует размер контекстного окна.
