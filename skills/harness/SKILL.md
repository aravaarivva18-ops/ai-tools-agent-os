# Harness-Solo — Team-Free Architecture & Skill Generator (v10)

Этот навык адаптирует концепцию **Harness (Team-Architecture Factory)** под ограничения **Strict Solo Loop** (без субагентов). Он позволяет автоматически генерировать плоскую структуру JIT-навыков, вспомогательных скриптов, тестов и конфигураций под новые доменные задачи, сохраняя кодовую базу чистой и компактной (YAGNI).

---

## 🏛️ Главные принципы
1. **Никаких субагентов (Strict Solo Loop)**: Harness-Solo категорически запрещает генерировать команды параллельных агентов (`TeamCreate`, `.claude/agents/`). Вместо этого вся логика пишется для **Единого Агента (Solo Agent)**, который использует JIT-навыки последовательно.
2. **Lean & YAGNI**: Максимум 2 уровня abstraction. Не плодить лишних файлов. Каждый новый навык должен решать конкретную прикладную задачу (например, парсинг, генерация PDF, отправка уведомлений).
3. **Регистрация в CLAUDE.md**: Все сгенерированные навыки должны быть зарегистрированы в [CLAUDE.md](file:///Users/rus/ai-tools/CLAUDE.md) в виде кратких поинтеров (триггерные правила + история изменений).

## 🔄 Пошаговый воркфлоу

### Phase 0: Аудит и проверка дублирования (De-duplication)
Перед созданием новой автоматизации обязательно просканируйте папку `skills/` и прочитайте [CLAUDE.md](file:///Users/rus/ai-tools/CLAUDE.md). 
* Если похожий навык уже существует (например, `stealth-scraping` для парсинга), расширяйте его, а не создавайте новый.
* Если домен пересекается с существующими бизнес-модулями (`geo-seo`, `ai-sales`, `ai-marketing`), новый навык должен ссылаться на них.

### Phase 1: Доменный анализ
1. Изучите стек технологий домена (библиотеки, базы данных, внешние API).
2. Выделите ключевые шаги выполнения задачи (сбор данных, логика, UI/вывод).
3. Используйте `Context7 MCP` для загрузки свежей документации по всем используемым библиотекам.

### Phase 2: Проектирование JIT-навыков
Определите, какие новые навыки требуются. Каждый навык должен быть герметичным и отвечать за один логический слой:
* `skills/<имя-навыка>/SKILL.md` — файл инструкций.
* `skills/<имя-навыка>/scripts/` — автоматические скрипты (например, парсеры, конвертеры).
* `skills/<имя-навыка>/tests/` — легкие unit-тесты (запуск <100мс).

### Phase 3: Генерация навыков (по стандарту `skill-creator`)
Сгенерируйте файлы навыков, следуя шаблону:
* **YAML Frontmatter**: Обязательные поля `name` и `description`. Описание должно быть максимально конкретным («pushy»), чтобы чётко триггерить вызов навыка в будущих сессиях.
* **Lean-документирование**: Ограничивайте `SKILL.md` до 500 строк. Все тяжелые справочные материалы, примеры API и логи выносите в подпапку `references/`.
* **Типобезопасность**: Все скрипты внутри `scripts/` должны возвращать структурированные данные, проходящие валидацию через Pydantic-модели из [tools/tool_validator.py](file:///Users/rus/ai-tools/tools/tool_validator.py).

### Phase 4: Настройка одноконтурной оркестрации
Опишите в файле `implementation_plan.md` пошаговый сценарий решения задачи для Solo-агента:
* Какой JIT-навык подключается на каждом этапе.
* Где сохраняются промежуточные данные (рекомендуется папка `_workspace/` внутри домена).
* Как контролировать KPI (время работы, отклик UI <2.0с).

### Phase 5: Регистрация поинтеров
Зарегистрируйте новые навыки в [CLAUDE.md](file:///Users/rus/ai-tools/CLAUDE.md) и [AGENTS.md](file:///Users/rus/ai-tools/tools/AGENTS.md). 
Укажите:
* Имя навыка и его триггерные ключевые слова.
* Дату создания и цель в таблице "История изменений" (변경 이력).

### Phase 6: Тестирование и валидация
Проведите проверку сгенерированной структуры:
1. Запустите тесты скриптов из папки `skills/<имя-навыка>/tests/` через `pytest`.
2. Проверьте триггеры вызова навыка (Should-trigger и Should-NOT-trigger сценарии).
