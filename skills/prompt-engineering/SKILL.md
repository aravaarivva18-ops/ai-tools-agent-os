# Разработка Промптов и Управление Контекстом (Prompt Engineering)

Этот навык описывает правила проектирования системных инструкций, структурирования контекста и оптимизации запросов для ускорения работы ИИ-агентов.

---

## 🏛️ 1. Структурирование контекста с помощью XML-тегов

Использование XML-тегов обеспечивает четкое разделение инструкций, примеров и динамических данных, предотвращая путаницу и prompt injection (внедрение вредоносных промптов через пользовательский ввод).

### Рекомендованный шаблон разметки:
```xml
<system_instructions>
Вы — эксперт по анализу кода. Ваша задача — находить ошибки в синтаксисе.
</system_instructions>

<examples>
  <example>
    <input>print "Hello"</input>
    <output>Ошибка: пропущены круглые скобки в вызове print.</output>
  </example>
</examples>

<project_context>
{{Файлы кодовой базы, переданные скрапером}}
</project_context>

<user_query>
{{Запрос пользователя}}
</user_query>
```

## ⚡ 2. Оптимизация промпт-кэширования (Prompt Caching)

При работе с большими языковыми моделями (например, Claude 3.5 Sonnet) правильное расположение блоков в запросе критично для попадания в кэш API:

1. **Статический блок (Всегда в начале):** Системные инструкции, схемы вывода (JSON schemas), описание инструментов (Tools) и few-shot примеры должны располагаться на самых первых строчках. Помечайте их флагом `cache_control: {"type": "ephemeral"}`.
2. **Динамический блок (Всегда в конце):** Код текущего файла, история переписки и непосредственный вопрос пользователя должны находиться в самом конце. Это гарантирует, что 90% контекста (который статичен) будет прочитан из быстрого кэша со скидкой до 90%.

## 🧠 3. Инструкции выполнения задач (CoT & ReAct)

Для повышения логики и качества вывода модели системные промпты должны обязывать её использовать логические цепочки:

*   **Chain of Thought (CoT - Ход мыслей):** Промпт должен требовать от модели сначала расписать свои рассуждения по шагам внутри тега `<scratchpad>` (черновик) перед тем, как выдать финальный ответ пользователю.
*   **Reasoning-Act-Observe (ReAct):** При работе с внешними инструментами агент должен строго следовать циклу:
    1.  **Thought (Мысль):** *«Зачем мне вызывать этот инструмент?»*
    2.  **Action (Действие):** Вызов инструмента (команды).
    3.  **Observation (Наблюдение):** Анализ возвращенного результата.

---

## 📐 4. Правило «200 строк» для инструкций проекта (`CLAUDE.md`)

*   **Размытие правил (Rule Dilution):** Чем больше размер файла правил проекта (таких как `CLAUDE.md` или `.cursorrules`), тем хуже модель воспринимает указания, находящиеся ближе к концу.
*   **Стандарт объема:** Файл `CLAUDE.md` должен быть компактным — **от 60 до 200 строк**.
*   **Исключение общеизвестных фактов:** Не описывайте в файлах правил общеизвестные синтаксические стандарты языков программирования (например, как работает TypeScript или React). Описывайте только особенности вашего проекта, локальный стек запуска и конкретные правила форматирования.
