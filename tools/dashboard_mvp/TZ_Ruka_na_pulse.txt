Техническое задание на разработку аналитического дашборда «Рука на пульсе» для агентства Target Media
Версия документа: для передачи исполнителю
1. Цель проекта
Необходимо разработать аналитический дашборд для агентства Target Media, который позволит в одном месте видеть ключевые показатели по клиентским проектам, быстро находить проблемные зоны и контролировать выполнение планов.
Дашборд нужен в первую очередь как внутренний управленческий инструмент для руководителя, аккаунт-менеджеров и проектной команды.
Основные задачи дашборда:
ежедневный контроль текущих результатов по проектам;
сравнение фактических показателей с плановыми;
выявление отклонений по лидам, стоимости лида, расходу бюджета и другим KPI;
сокращение ручной подготовки отчетов;
формирование единого окна контроля по всем клиентам агентства.
2. Общая логика и этапность разработки
Разработка должна вестись поэтапно, чтобы сначала запустить рабочий MVP на основных источниках данных, а затем постепенно наращивать функциональность.
Этап
Источники данных
Что должно быть реализовано
Этап 1 — MVP
Яндекс.Директ; Яндекс.Метрика; Google Таблицы
Сводный экран по всем клиентам; карточка клиента; план vs факт; базовые графики; ручной лог изменений через Google Таблицы; фильтры по клиенту, периоду и проекту.
Этап 2 — расширение
VK Реклама; Авито Ads; Яндекс Карты / 2ГИС; SEO; Callibri; amoCRM
Подключение дополнительных источников; данные по звонкам; CRM amoCRM; возможность заводить медиапланы, KPI и внутренние данные напрямую внутри дашборда; более глубокая детализация по качеству лидов и этапам воронки.
Этап 3 — развитие системы
Telegram Ads; другие системы коллтрекинга; Битрикс24
Дополнительные интеграции; развитие логики сквозной аналитики; продвинутая отчетность и расширение автоматизации.

Первый этап:

4. Понятие лида и конверсии
Так как у разных клиентов логика учета лидов может отличаться, необходимо предусмотреть гибкую настройку конверсий по каждому проекту.
Возможные типы лидов:
отправка формы на сайте;
достижение цели в Яндекс.Метрике;
звонок;
сообщение;
создание сделки или лида в CRM;
квалифицированный лид по определенному этапу в CRM.
Необходимо предусмотреть возможность отображения как минимум двух уровней конверсий:
Первичный лид.
Квалифицированный лид.
5. Плановые и фактические показатели
По каждому клиенту / проекту необходимо отображать:
плановый бюджет;
фактический расход;
плановое количество лидов;
фактическое количество лидов;
плановую стоимость лида;
фактическую стоимость лида.
Если на следующих этапах будут доступны соответствующие данные, желательно предусмотреть возможность отображения:
выручки;
квалифицированных лидов;
стоимости квалифицированного лида;
ROMI / ROI.
Дашборд должен уметь сравнивать:
план vs факт;
текущий период vs предыдущий период;
текущую динамику vs план на месяц.
6. Учет расходов
Необходимо предусмотреть логику учета расходов таким образом, чтобы можно было отображать:
чистые рекламные расходы;
расходы с учетом НДС;
расходы с учетом НДС и агентской комиссии.
Желательно, чтобы способ расчета можно было задавать гибко, в зависимости от проекта.
7. Структура доступа к данным
Необходимо предусмотреть, что у агентства могут быть разные схемы доступа:
Архитектура решения должна поддерживать оба варианта.
часть клиентов находится в агентских кабинетах;
часть клиентов находится в личных кабинетах клиентов, где агентству выданы доступы.
8. Масштабируемость
Система должна проектироваться не под текущее количество клиентов, а с расчетом на дальнейший рост агентства.
Важно, чтобы:
новый клиент подключался без полной переделки системы;
логика работы строилась через единые справочники и правила;
дашборд можно было масштабировать на десятки проектов.
9. Основные экраны дашборда
9.1. Сводный экран по всем клиентам
На главном экране руководитель должен видеть:
список всех активных клиентов / проектов;
основные KPI по каждому проекту;
выполнение плана;
проблемные проекты;
отклонения по CPL;
отклонения по лидам;
отклонения по расходу бюджета.
9.2. Детальная карточка клиента
В карточке клиента необходимо отображать:
основные показатели по проекту;
детализацию по источникам;
план vs факт;
динамику по дням / неделям / месяцам;
лог изменений;
при необходимости — комментарии и ручные пояснения.
10. Основные KPI
Минимально нужно отображать:
расход бюджета;
показы;
клики;
CTR;
CPC;
количество лидов;
CPL.
На следующих этапах или при наличии данных:
квалифицированные лиды;
стоимость квалифицированного лида;
выручка;
ROMI / ROI;
показатели CRM-воронки;
данные по звонкам;
данные по картам и SEO.
11. Лог изменений
На первом этапе лог изменений реализуется вручную через Google Таблицы.
Формат записи:
дата;
клиент / проект;
описание изменения;
комментарий / причина;
ожидаемый эффект.
На следующих этапах можно рассмотреть автоматизацию части логов, если это будет технически целесообразно.
12. Графики и визуализация
На первом этапе необходимы базовые графики:
динамика расхода;
динамика лидов;
динамика CPL.
В дальнейшем можно расширять:
динамикой квалифицированных лидов;
динамикой звонков;
динамикой по CRM-этапам;
динамикой по картам и SEO.
13. Сценарий использования
13.1. Руководитель агентства
Заходит в дашборд и быстро видит:
какие проекты в норме;
какие проекты требуют внимания;
где просели лиды;
где вырос CPL;
где идет перерасход бюджета;
насколько выполняется план.
13.2. Аккаунт-менеджер / PM / специалист
Переходит в карточку клиента и анализирует:
текущую динамику;
отклонения;
последние изменения;
причины проблем;
точки для принятия решений.
14. Пользователи и доступы
На первом этапе дашборд предназначен для внутреннего использования сотрудниками агентства:
руководство;
аккаунт-менеджеры;
project managers;
специалисты.
На следующих этапах желательно предусмотреть возможность гостевого доступа для клиентов, при котором клиент видит только свои данные.
15. Брендинг
Необходимо предусмотреть брендирование дашборда:
логотип агентства;
аккуратное оформление в стиле White Label.
16. Частота обновления данных
На первом этапе желательно реализовать обновление данных по расписанию.
Предпочтительный режим:
автоматическое обновление каждые 2–4 часа в рабочее время;
либо иной согласованный интервал, который не будет упираться в ограничения API;
возможность ручного обновления данных.
17. Дополнительные пожелания
цветовая индикация отклонений;
фильтры по периоду, клиенту, проекту, источнику;
возможность дальнейшего подключения Telegram-уведомлений;
возможность сравнения периодов;
возможность прогноза до конца месяца на последующих этапах;
система справочников для масштабируемого подключения новых клиентов.
18. Что является результатом первого этапа
Результатом первого этапа должен стать рабочий MVP-дашборд, который:
автоматически забирает данные из Яндекс.Директ;
автоматически забирает данные из Яндекс.Метрики;
использует Google Таблицы для ручных и плановых данных;
показывает сводку по агентству;
показывает детальную карточку клиента;
позволяет сравнивать план и факт;
дает базовую динамику по ключевым показателям;
позволяет вести ручной лог изменений.
19. Что не является задачей первого этапа
Чтобы избежать лишних ожиданий, на первом этапе не требуется:
подключение всех рекламных систем сразу;
подключение всех CRM и коллтрекингов;
полная сквозная аналитика;
сложная финансовая модель;
автоматический лог изменений из всех систем;
полная клиентская версия кабинета.
Это относится к следующим этапам развития продукта.
20. Ключевой принцип разработки
Важно, чтобы решение изначально строилось не как разовый красивый отчет, а как масштабируемая аналитическая система для агентства, которую можно развивать поэтапно без пересборки с нуля.
