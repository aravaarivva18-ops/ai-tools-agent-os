# Отчет по аудиту и улучшению производительности ai-tools

## 1. Замеры времени импорта CLI
- **Замер**: `python -X importtime -c "import tools.cli"`
- **Cumulative time**: **29.3 ms** (чистое время загрузки CLI и базовых системных модулей).
- **Результаты анализа**:
  - Все тяжелые сторонние библиотеки (`fastembed`, `tiktoken`, `google-genai`, `json_repair`) импортируются строго лениво (внутри функций вызова).
  - На старте импортируются только стандартные системные пакеты (`os`, `sys`, `pathlib`, `subprocess`, `argparse`).
- **Вердикт**: Время импорта идеальное, дополнительные оптимизации импортов CLI не требуются.

## 2. Анализ токенизации и сжатия контекста в SoloLoopV10

### Проблема
Текущая функция `compact_context` обрезает логи по фиксированному числу строк (`max_lines=5`). Это не гарантирует укладывание в лимит токенов (например, 2000 токенов) при накоплении шагов в истории.

### Решение
Разработать и внедрить алгоритм **точного сжатия истории (Exact Token Compression)** на базе `count_tokens_exact` из `tools/context_utils.py`:
1. Вычислить точные токены для каждого шага: `tokens = count_tokens_exact(command) + count_tokens_exact(output)`.
2. Если суммарные токены всех шагов превышают лимит (`max_tokens=2000`):
   - **Фаза 1 (Схлопывание)**: Начиная с самых старых успешных шагов, обрезать их `output` до строки `"[success output trimmed]"` (сохраняя сам факт успешного запуска).
   - **Фаза 2 (Прунинг)**: Если сумма токенов все еще превышает лимит, полностью удалять самые старые шаги из истории, пока размер не впишется в лимит.
3. Этот подход гарантирует безопасность контекста при сохранении истории запусков.

## 3. Анализ RAG-поиска и Search Cache

### Проблема
Каждый запуск RAG-поиска выполняет повторное чтение и десериализацию JSON-индекса (`handoffs_index.json`), а при векторном поиске — инициализацию модели FastEmbed и генерацию эмбеддинга для запроса. Это замедляет работу на ~1.5–2 секунды при каждом вызове.

### Решение
Внедрить легковесный **Search Cache** для запросов к базе знаний:
1. Создать локальный файл `vault/search_cache.json` для сохранения результатов.
2. Структура кэша:
   ```json
   {
     "queries": {
       "query_text": {
         "mtime": 1719912000.0,
         "brief": "Текст сгенерированного дайджеста"
       }
     }
   }
   ```
3. При вызове `get_semantic_brief(query)`:
   - Считать `mtime` папки `vault/handoffs/` (время последнего изменения файлов).
   - Если в кэше есть этот `query` и его `mtime` совпадает с текущим `mtime` папки — возвращать `brief` из кэша мгновенно (за 1–2 мс).
   - Если кэш устарел или запроса нет — выполнять поиск, сохранять результат в кэш и возвращать.
